# IT 포트폴리오 제작 AI Agent 가이드 지침서

> **문서 목적**: AI Agent가 IT 직무 포트폴리오를 제작/검수할 때 따라야 할 표준 작성 지침
> **출처**: SSAFY 취업지원센터 「IT포트폴리오 작성 가이드북」 기반
> **대상 직무**: 프론트엔드 / 백엔드 / 임베디드 / AI·Data 엔지니어

---

## 0. AI Agent 최상위 원칙 (반드시 선행 확인)

포트폴리오를 생성·수정하기 전, AI Agent는 다음 4가지를 **먼저** 파악해야 한다.

1. **채용 직무**: 지원 기업 및 직무(프론트/백엔드/임베디드/AI)
2. **사용자 보유 기술**: 실제 구현 경험이 있는 기술 스택
3. **대표 프로젝트 1~2개**: 본인이 구현한 기능이 명확한 프로젝트
4. **수치화 가능한 성과**: Before/After 데이터, 사용자 수, 성능 지표 등

> ⚠️ **합격하는 포트폴리오의 대전제**: **"채용 직무와의 적합성"이 가장 중요하다.**
> 모든 콘텐츠는 지원 직무 관점에서 재구성되어야 한다.

---

## 1. 핵심 평가 3요소 (Core Evaluation Criteria)

인사담당자와 현업 개발자가 포트폴리오에서 **반드시** 보고자 하는 3가지 요소이다. AI Agent는 최종 산출물이 아래 3요소를 모두 충족하는지 검증해야 한다.

### 평가 요소 ①: 개인의 역할 및 기여도 (Role & Contribution)
- **문제점**: 팀 프로젝트에서 구현한 '전체' 기능을 나열하면 지원자의 역량 판별 불가
- **원칙**: 반드시 **"내가 담당한 부분"만** 명확히 분리해 작성
- **AI Agent 체크**: "팀이 구현했다" vs "내가 구현했다"를 구분했는가?

### 평가 요소 ②: 문제 해결 사례 (Problem-Solving Story)
- **문제점**: 기술적 어려움·트러블슈팅·디버깅 경험이 빠진 포트폴리오는 문제해결역량 평가 불가
- **원칙**: 반드시 **원인 분석 → 해결 방안 → Before/After 비교**를 포함
- **AI Agent 체크**: 최소 1건 이상의 문제 해결 스토리가 수치와 함께 작성되었는가?

### 평가 요소 ③: 기술 선정 이유 (Tech Selection Rationale)
- **문제점**: "왜 그 기술을 썼는가"에 대한 답변이 없으면 기술적 깊이 없음으로 판단
- **원칙**: 각 주요 기술마다 **프로젝트 특성과 연결된 선택 근거**를 제시
- **AI Agent 체크**: 모든 주요 기술에 "→ 선택 이유" 한 줄이 붙었는가?

## 2. 포트폴리오 표준 구조 (4 STEP)

AI Agent는 아래 4단계 구조를 **순서대로** 생성해야 한다.

```
STEP 1. 프로필 (Profile)
 ├─ 1-1. 슬로건 + 한줄 소개
 ├─ 1-2. 기술 스킬
 └─ 1-3. 교육/학습 이력

STEP 2. 프로젝트 경험 (Project Experience)
 ├─ 2-1. 프로젝트 개요
 ├─ 2-2. 담당 역할 및 기여도
 ├─ 2-3. 사용 기술 및 선택 이유
 └─ 2-4. 구현 기능

STEP 3. 프로젝트 성과 및 결과 (Results)
 ├─ 3-1. 문제 해결 사례
 ├─ 3-2. 성과 및 결과
 └─ 3-3. 프로젝트 회고

STEP 4. 직무별 특화 요소 (Job-specific Tips)
```

## 3. STEP 1 — 프로필 작성 지침

### 3-1. 슬로건 + 한줄 소개

#### 작성 원칙
- **직무와 연관된 나만의 강점**을 활용할 것
- **구체적인 수식어**를 사용할 것 (추상적 단어 금지)
- 슬로건: 차별화된 역량 한 문장
- 한줄 소개: 개발자로서의 가치관, 중장기 목표, 현재 가장 깊이 있게 구현 중인 스킬

#### ❌ Worst 예시 (사용 금지)
- "끊임없이 학습하는 개발자"
- "도전하는 개발자"
- "개발을 좋아하는 개발자"
- "꾸준히 성장하는 개발자"
- "새로운 기술 학습을 좋아합니다"
- "소통을 잘합니다"

> **이유**: 누구에게나 해당되는 의미 없는 수식어. 어떤 역량을 보유했는지 파악 불가.

#### ✅ Best 예시 (권장 패턴)
- 슬로건: **"안정적이고 확장 가능한 서비스를 설계하는 백엔드 개발자 OOO입니다."**
- 한줄 소개: "사용자 경험의 기반은 튼튼한 백엔드라고 생각합니다. 데이터 흐름을 안정적으로 관리하고, 효율적인 서버 아키텍처와 최적화된 API를 제공하는 것이 백엔드 개발자의 핵심 역할이라고 믿습니다. 저는 프로젝트 경험을 통해 성능 최적화, 보안 강화, 대용량 트래픽 대응에 집중해 왔으며, 앞으로도 확장성 있는 시스템을 설계하고 운영하는 백엔드 개발자로 성장하고자 합니다."

### 3-2. 기술 스킬 (SKILLS)

#### 작성 원칙
1. **스킬을 카테고리화**하여 제시 (Frontend / Backend / DB / AI 등)
2. **숙련도를 시각화** (막대그래프, 별점, 5단계 척도 등)
3. **주요 스킬은 활용 수준을 구체적으로 서술**
4. 희망 직무와 연관된 보유 기술 위주로 작성

#### 숙련도 5단계 기준 (표준안)
| 등급 | 기준 |
|------|------|
| ★☆☆☆☆ | 기본적 문법 이해, 간단 코드 작성 가능 |
| ★★☆☆☆ | 문서 참고하여 기능 구현, 간단한 프로젝트 참여 가능 |
| ★★★☆☆ | 독립적으로 개발, 문서 및 가이드 활용하여 최적화 가능 |
| ★★★★☆ | 프로젝트 내 핵심 개발 담당, 코드 리뷰 및 최적화 가능 |
| ★★★★★ | 기술 리더 역할 수행, 코드 리뷰/멘토링, 아키텍처 설계 가능 |

#### ❌ Worst 패턴
- 무조건 상(上) 또는 무조건 하(下)로 몰아서 표기
- 스킬 단순 나열: `JAVA, Spring Boot, JPA, MySQL, Redis, Docker`

#### ✅ Best 예시
```
SKILLS
JavaScript  ■■■■□
  - ES6+ 를 이해하고 이를 활용하여 소규모 프로젝트 진행
  - JS기반 프레임워크와 라이브러리 사용 가능

Vue.JS  ■■■■□
  - Life Cycle을 이해하고, Vuex, Vue-router를 활용하여 소규모 프로젝트 진행 가능
  - 컴포넌트 개념을 이해하고, 재사용을 위한 개발 가능
```

### 3-3. 교육/학습 이력

#### 작성 원칙
- 교육이 많을 경우 **직무와 관련된 교육 위주로** 선별
- 교육명·기간만 나열 금지. **구체적 기간 + 학습 시간 + 주요 커리큘럼 + 습득한 지식/기술**까지 서술
- **SSAFY에서만 사용하는 용어(공통/관통 프로젝트 등)는 외부 독자가 이해할 수 있게 풀어서 작성**

#### ❌ Worst 예시
```
한마음 ICT 멘토링   2024.05 ~ 2024.07
ICT 학점연계 프로젝트 인턴십  2024.09 ~ 2024.11
삼성청년SW·AI아카데미 00기  2025.07 ~ 재학 중
```
> 어떤 커리큘럼이었는지, 무엇을 습득했는지 알 수 없음.

#### ✅ Best 예시
```
AWARDS / CERTIFICATES
2024 우수상  대회/수상명
2023 최우수상  OO대학교 & OO전자 기업연계 프로젝트
2021 금상  OO자동차 주최 해커톤 대회

EDUCATION / EXPERIENCE
(2025 ~ 현재) 삼성청년SW·AI아카데미 14기 재학중
  - Java, SpringBoot, Vue 등 학습 및 프로젝트 적용
  - 애자일 기반의 프로젝트 협업 경험
  - Python 기반 머신러닝/딥러닝 기초 학습
  - 텍스트 데이터 분석 프로젝트 경험
(2024 ~ 2025) OO전자 인턴십 수료
(2020 ~ 2024) OO대학교 컴퓨터공학과 졸업
```

## 4. STEP 2 — 프로젝트 경험 작성 지침

### 4-1. 프로젝트 개요

#### 작성 원칙
- 프로젝트의 **배경 → 목표 → 주요 기능** 순으로 서술
- 주요 기능은 **키워드 중심**으로 간략히
- 해당 서비스를 **직관적으로 이해**할 수 있게 작성

#### ❌ Worst 패턴
- 개요를 아예 작성하지 않음
- "책을 추천하는 프로그램 개발" 같은 막연한 한 줄

#### ✅ Best 예시
```
프로젝트명: Edu Learn 플랫폼 (2025.07 ~ 2025.08 / 인원 5명)

□ 프로젝트 개요
 - 목표 : 학생과 강사를 연결해 맞춤형 학습 콘텐츠 제공
 - 배경 : 개인화 학습 수요 증가에 따른 효율적 학습 지원 필요
 - 주요 기능 : 강의 검색 및 추천, 실시간 Q&A, 학습 진도 추적, 맞춤형 학습 리포트 제공
```

### 4-2. 담당 역할 및 기여도

#### 작성 원칙 (매우 중요 — 평가 1순위)
- **팀 내 역할 분담**을 명시 (전체 팀 구성 + 각자의 역할)
- **본인의 담당 역할**을 구체적으로 작성
- **업무 단위 / 프로젝트 단위**로 기여도를 **% 또는 구체 수치**로 표기
- **구체적으로 어떤 부분**에 기여했는지 서술

#### ❌ Worst 패턴
- "팀원들을 도와 프로젝트 수행"
- "분배 후 협력하여 완성"
- "프로젝트 전반적으로 기여"
- 구체적 기여 내용 없음

#### ✅ Best 예시
```
□ 담당 역할 (백엔드 개발자)
 - 서버 아키텍처 설계 : 학습 콘텐츠 관리 및 사용자 인증을 위한 구조 설계
 - API 개발 : 사용자 인증, 학습 진도 관리, 강의 자료 업로드/다운로드 REST API 구현
 - DB 최적화 : 쿼리 효율화, 인덱스 설계로 데이터 조회 성능 개선
 - 보안 강화 : JWT 기반 토큰 인증으로 사용자 데이터 보호

※ 팀 내 역할 분담
 - 프론트엔드 (2명) : React 기반 UI 설계 및 최적화
 - 백엔드 (2명) : 서버, DB 아키텍처 설계, API 개발 (본인 역할)
 - 데이터 분석 (1명) : 학습 데이터 기반 추천 알고리즘 개발

□ 개인 기여도
 - API 설계 및 구현 : 35%
   → 사용자 인증, 학습 진도 관리, 자료 업로드/다운로드 API 개발
 - DB 설계 및 최적화 : 20%
   → 테이블 구조 설계, 인덱스 최적화, 쿼리 성능 개선
 - 파일 관리 기능 구현 : 15%
   → AWS S3 연동 업로드/다운로드 모듈 개발
 - 보안 강화 : 10%
   → JWT 기반 인증·인가 처리 설계
 - 데이터 분석 지원 : 10%
   → 학습 진도 데이터 제공 API, 추천 알고리즘 적용 협업
 - 프로젝트 관리 및 협업 : 10%
   → JIRA/Slack 기반 일정 관리, 문서화, 팀 내 코드 리뷰
```

### 4-3. 사용 기술 및 선택 이유 (평가 3순위)

#### 작성 원칙
- 여러 기술 나열 금지. **프로젝트에서 실제 사용한 주요 기술**만
- 각 기술마다 **"왜 선택했는가"**를 반드시 작성
- **프로젝트 특성과 기술 장점**을 연결

#### ❌ Worst 패턴
- 기술만 나열하고 선택 이유 없음
- "명세서에 제시되어 선택" 등 수동적 선택

#### ✅ Best 예시
```
□ 사용 기술 / 적용 맥락 및 이유
- Java & SpringBoot
  → REST API 안정적 제공과 확장 가능한 구조 설계에 적합. (Spring Security로 보안 강화 기능)
- MySQL
  → 학습 진도·로그·강의 데이터 등 관계형 구조 처리에 효율적. 인덱스 최적화로 성능 개선
- React
  → 사용자 친화적인 학습 리포트 UI 제공을 위해 사용 (프론트엔드 팀원과 협업)
- AWS S3 + CloudFront
  → 대용량 강의 자료 저장/배포 문제 해결. S3로 스토리지 관리, CloudFront로 CDN 분산 처리
- Redis
  → 반복되는 파일 다운로드 요청을 캐싱해 서버 부하 완화 및 응답 속도 향상
```

### 4-4. 구현 기능

#### 작성 원칙
- **내가 구현한 핵심 기능만** 작성 (프로젝트 전체 기능 설명 금지)
- 기능별로 **기술 적용 내역**을 괄호/하위 항목으로 명시

#### ✅ Best 예시
```
□ 구현 사항
[학습 진도 추적 API]
 - 학생별 학습 진행 상태를 DB에 저장
 - REST API 형태로 데이터 제공 → 프론트엔드에서 시각화 및 리포트 생성 가능

[강의 자료 업로드/다운로드 기능]
 - AWS S3 연동을 통한 대용량 파일 안정적 저장 및 관리
 - CloudFront 적용으로 빠른 다운로드 성능 확보

[JWT 기반 사용자 인증/인가 시스템]
 - 로그인/회원가입 API 구현
 - JWT 토큰 기반 인증 절차 설계 → 세션 기반 인증 대비 보안 강화

[DB 최적화 및 구조 설계]
 - 자주 조회되는 학습 진도, 강의 정보에 인덱스 적용
 - 정규화된 구조와 일부 캐싱 구조(조회용 테이블) 병행 적용

[Redis 캐싱 전략 도입]
 - 자주 요청되는 강의 자료 메타데이터를 캐시에 저장
 - 평균 응답 속도 40% 이상 단축

[추천 알고리즘 데이터 제공 API (AI 협업 지원)]
 - Word2Vec 기반 추천 모델과 연계할 수 있도록 학습 로그 데이터 가공 후 제공
```

## 5. STEP 3 — 프로젝트 성과 및 결과 작성 지침

### 5-1. 문제 해결 사례 (평가 2순위 — 매우 중요)

#### 작성 원칙
- **발생한 문제**를 명확히 서술
- **해결 방법 + 선택 이유 + 결과**를 작성
- **원인 분석**과 **해결방법 선택 이유**를 자세히
- **Before & After 비교**를 정확한 수치로 제시

#### ❌ Worst 패턴
- "캐싱 전략으로 문제 해결" (과정 생략)
- "팀원과 소통하여 문제 해결" (기술적 해결이 아님)

#### ✅ Best 예시 (표준 템플릿)
```
□ 문제 해결 사례

1. 오류 분석
 - 문제 현상 : 대용량 강의 자료 다운로드 시 서버 과부하 및 응답 지연 발생 (평균 5~7초)
 - 원인 : 동일 파일 반복 요청이 서버 리소스 과도 사용 → 성능 저하 발생
 - 추가 확인 사항 : 피크 시간대 CPU 사용률 85%까지 상승

2. 개선 과정
 [문제 인식] → [로그 분석] → [캐싱 전략 설계] → [Redis 적용] → [AWS CloudFront 적용] → [성능 검증]

3. Before & After 비교
 | 지표         | Before       | After                |
 |--------------|--------------|----------------------|
 | 응답 속도    | 5~7초        | 2~3초 (약 60% 단축) |
 | 서버 CPU     | 최대 85%     | 최대 55% (30% 개선) |
 | 피크 시간대  | 지연 다수 발생 | 안정적 서비스 제공   |

4. 해결 과정 서술
 1) 문제 진단 : 로그 분석으로 반복 요청 확인
 2) 전략 수립 : Redis 캐싱 구조 설계로 요청 분산 처리
 3) 기술 적용 : Redis + AWS S3 + CloudFront 통합 적용
 4) 검증 : JMeter 성능 테스트 결과 → 개선 효과 확인
```

### 5-2. 프로젝트 성과 및 결과

#### 작성 원칙
- **전/후 수치 변화**를 통해 증명
- **시각화·데이터화**가 핵심
- 엄청난 성과가 아니어도 됨. **협업 도구 활용한 시간 단축 등 정성적 성과**도 가능

#### ❌ Worst 패턴
- "OO팀원과 적극적인 소통"
- "문제 해결 후 속도 상승"
- 구체적 수치·결과 없음

#### ✅ Best 예시
```
□ 프로젝트 성과 및 결과
 - 사용자 수 증가 : SSAFY 교육생에게 배포 후 3개월 내 사용자 수 500명 돌파
 - 강의 수강률 개선 : 학습 진도 추적 기능 도입 후, 맞춤형 학습 리포트 제공으로
   강의 수강 완료율이 45%에서 70%로 상승
 - 개발 기간 단축 : JIRA와 Slack을 통한 체계적인 작업 관리 및 원활한 커뮤니케이션으로
   프로젝트 일정을 2주 앞당겨 성공적으로 마무리할 수 있었음
 - SSAFY 프로젝트 우수상 수상
 - 프로젝트 결과물과 기술적 접근 방법을 우수 사례로써 타 팀에 공유하여 기술 전파
```

### 5-3. 프로젝트 회고

#### 작성 원칙
- **배운 점 / 깨달은 점**을 작성
- **아쉬웠던 부분 + 개선 방안**을 제시하면 더 좋음
- 회고 역시 **구체적으로** 작성

#### ❌ Worst 패턴
- "시간 부족으로 모든 기능 구현 X"
- "팀워크의 중요성을 깨달았다"

#### ✅ Best 예시
```
□ 프로젝트 회고

[아쉬웠던 부분]
 - 학습 진도 데이터가 쌓일수록 조회 성능 저하 발생
   → 초기 DB 스키마 설계에서 데이터 증가량을 충분히 고려하지 못함
 - Redis 캐싱 구조가 단일 서버 기반이라 장애 시 대응 플랜 부족
   → 분산 캐싱 전략 미흡

[개선 방안]
 - DB 파티셔닝/샤딩 도입 검토하여 대규모 데이터 확장성 보장
 - Redis Sentinel 또는 Cluster 기반 분산 캐싱 적용으로 장애 대응력 강화
 - ELK Stack(Logstash, Kibana) 기반 로그 분석 환경 구축해 성능 모니터링 자동화

[학습 결과]
 - 단순 기능 구현을 넘어, 성능·확장성·안정성을 고려한 아키텍처 설계의 중요성 체감
 - 캐싱, CDN, 보안 인증 등 백엔드 핵심 기술이 실제 서비스 품질 개선에 직접적으로 기여한다는 점을 경험

[깨달은 점]
 - 백엔드 개발자의 역할은 단순히 API를 만드는 것이 아니라,
   사용자가 끊김 없는 안정적 서비스를 경험하도록 시스템을 설계하고 운영하는 것임을 알게 됨
 - 성능 테스트, 장애 대응 시나리오, 데이터 관리 전략을 프로젝트 초반부터 고려해야 한다는 점을 배움
```

## 6. STEP 4 — 직무별 특화 작성 가이드

AI Agent는 사용자의 **지원 직무**를 확인한 뒤, 아래 해당 섹션의 요소를 **반드시 포함**해야 한다.

### 6-1. 프론트엔드

| 강조 요소 | 포함할 내용 |
|-----------|-------------|
| **비주얼 & UI/UX** | 프로젝트 화면 캡처, UI/UX 개선 과정, 사용자 경험 고려한 디자인 적용 사례와 피드백 반영 과정 |
| **반응형 웹 & 성능 최적화** | 미디어 쿼리, Flexbox, Grid 활용 / 코드 스플리팅, Lazy Loading, 이미지 최적화 사례 |
| **상태 관리 & 데이터 처리** | React Query 활용한 API 성능 개선, WebSocket 기반 실시간 처리, 클라이언트 캐싱 경험 |
| **가독성** | 글꼴, 여백 점검 필수 (타 직무 대비 가독성 중요도 높음) |

### 6-2. 백엔드

| 강조 요소 | 포함할 내용 |
|-----------|-------------|
| **아키텍처 설계 능력** | API 설계(RESTful, GraphQL), DB 모델링(정규화, 관계형/비관계형 선택), MVC·클린 아키텍처·MSA 설계 시각화 |
| **성능 최적화 & 확장성** | 캐싱(Redis, Memcached), 로드 밸런싱, 트랜잭션 처리, N+1 문제 해결, 인덱스 적용, Docker/Kubernetes 활용 |
| **테스트 코드 & 보안** | 단위/통합/E2E 테스트, JWT, OAuth, HTTPS 적용, 보안 취약점 해결 경험 |

### 6-3. 임베디드

| 강조 요소 | 포함할 내용 |
|-----------|-------------|
| **하드웨어 연동 경험** | 센서, MCU, 디바이스 드라이버 활용 사례 / UART, I2C, SPI 등 통신 방식 처리 |
| **성능 & 전력 관리** | 저전력 모드, 메모리 최적화, RTOS 적용 사례 / 타이머 최적화, 인터럽트 처리 수치 |
| **회로도 & 데이터시트** | 개발한 시스템의 회로도, 부품 데이터시트 분석 내용, 실제 이미지/작동 영상, Github 연동 |

### 6-4. AI / Data

| 강조 요소 | 포함할 내용 |
|-----------|-------------|
| **모델 성능 & 평가 지표** | Accuracy, Precision, Recall 등 지표 / 모델 개선을 위해 적용한 기법과 그 효과 |
| **데이터 전처리 & 파이프라인** | 결측치 처리, 이상치 제거, 스케일링 + 코드 / 수집→처리→학습 자동화 파이프라인 |
| **실제 적용 사례** | AI 모델이 실제 서비스에서 활용된 사례, 개선된 지표, Confusion Matrix / ROC Curve 등 시각화 |

## 7. AI Agent 최종 검증 체크리스트

포트폴리오 생성 완료 후, AI Agent는 아래 항목을 **모두 통과**하는지 검증해야 한다.

### ✅ 전체 구조 검증
- [ ] 4개 STEP(프로필 / 프로젝트 경험 / 성과·결과 / 직무별 요소) 모두 포함되었는가?
- [ ] 지원 직무와의 적합성이 전체 문서 전반에 드러나는가?

### ✅ STEP 1 (프로필) 검증
- [ ] 슬로건에 추상적 단어(성장, 도전, 학습, 소통 등)만 사용되지 않았는가?
- [ ] 한줄 소개에 직무 관련 가치관/목표가 구체적으로 담겼는가?
- [ ] 기술 스킬이 카테고리화 + 숙련도 시각화되었는가?
- [ ] 주요 스킬마다 활용 수준이 서술되었는가?
- [ ] 교육 이력에 커리큘럼과 습득 기술이 구체적으로 기재되었는가?
- [ ] SSAFY 내부 전용 용어(공통/관통 프로젝트 등)를 외부인이 이해할 수 있게 풀어 썼는가?

### ✅ STEP 2 (프로젝트 경험) 검증
- [ ] 프로젝트 개요가 배경/목표/주요 기능 구조로 작성되었는가?
- [ ] 팀 구성과 역할 분담이 명시되었는가?
- [ ] **본인의 역할과 기여도가 %/수치로 구체화**되었는가?
- [ ] "팀의 구현" vs "나의 구현"이 명확히 분리되었는가?
- [ ] **모든 주요 기술에 선택 이유(→ …) 가 붙어 있는가?**
- [ ] 구현 기능이 "내가 구현한 것"만으로 제한되었는가?

### ✅ STEP 3 (성과 및 결과) 검증
- [ ] **최소 1건 이상의 문제 해결 사례**가 있는가?
- [ ] 문제 해결 사례에 "오류 분석 → 개선 과정 → Before/After → 해결 과정"이 모두 포함되었는가?
- [ ] **Before/After가 수치 비교표로 제시**되었는가?
- [ ] 성과가 정량/정성 지표로 표현되었는가?
- [ ] 회고에 아쉬운 점 + 개선 방안 + 학습 결과가 구체적으로 작성되었는가?

### ✅ STEP 4 (직무별 특화) 검증
- [ ] 지원 직무에 해당하는 특화 요소가 3개 이상 반영되었는가?
- [ ] 직무별 핵심 키워드(백엔드=아키텍처·성능·보안 / 프론트=UI·반응형·상태관리 등)가 등장하는가?

### ✅ 금지 표현 필터 (최종 점검)
아래 표현이 남아있지 않은지 확인:
- "끊임없이 학습하는", "도전하는", "꾸준히 성장하는", "개발을 좋아하는"
- "새로운 기술 학습을 좋아합니다", "소통을 잘합니다"
- "팀원들을 도와", "분배 후 협력하여 완성", "프로젝트 전반적으로 기여"
- "책을 추천하는 프로그램", "개요 미작성"
- "캐싱으로 해결" (과정 생략), "팀원과 소통하여 해결" (기술적 해결 아님)
- "시간 부족", "팀워크 중요성" 수준의 얕은 회고

## 8. 면접 대비 원칙 (포트폴리오 작성 시 반드시 인지)

> **"포트폴리오에 쓴 내용은 전부 직무 면접 질문 대상이다."**

AI Agent는 포트폴리오 생성 시, 사용자가 **자신이 쓴 내용 전부에 대해 면접에서 답변할 수 있어야 함**을 고려해야 한다.

- 기술을 나열만 하는 것이 아니라 **"왜 썼는지, 어떻게 썼는지"**를 답할 수 있는 수준으로 작성
- 문제 해결 사례는 **원인 분석 논리**를 설명할 수 있어야 함
- 수치는 **측정 방법**까지 답할 수 있는 근거가 있어야 함

사용자가 직접 구현하지 않은 기능, 직접 경험하지 않은 문제 해결은 **절대 포함하지 않는다.**

## 9. 최종 결론

> ★ **합격하는 포트폴리오의 공식** ★
>
> **채용 직무와의 적합성** × **개인 역할/기여도 명확성** × **문제 해결 스토리** × **기술 선정 이유**
>
> = 4가지가 모두 충족될 때 통과하는 포트폴리오가 완성된다.

AI Agent는 사용자의 **실제 경험**을 기반으로, 위 지침에 따라 **면접에서도 당당히 설명할 수 있는** 포트폴리오를 제작해야 한다.
